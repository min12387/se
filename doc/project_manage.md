# 📄 프로젝트 관리 계획서

---

## 1. 개요 (Overview)

### 1.1 문서 목적
본 문서는 Mini Drive 시스템 개발을 체계적으로 수행하기 위해 프로젝트의 일정, 인력 구성, 기술 운영 방식, 품질 기준 및 위험 대응 전략을 정의하는 것을 목적으로 한다.

### 1.2 프로젝트 개요
Mini Drive는 조직 내 파일을 통합 관리하고 협업을 지원하는 클라우드 기반 파일 관리 시스템이다.  
대용량 파일 업로드/다운로드, 파일 공유 및 권한 설정, 검색 기능을 제공한다.  

**예시**
- 사용자 A가 파일 업로드 → 사용자 B에게 공유 → 권한 설정 → 검색 기능으로 조회

### 1.3 참조 문서
- Project Definition Document  
- Requirement Specification  
- Quality Attribute Document

## 2. 개발 계획 (Development Plan)

### 2.1 개발 방식
- Agile 기반 SCRUM 방식 적용  
- Sprint 단위 반복 개발 (2주 단위)

**SCRUM 구성**
- Sprint Planning  
- Daily Scrum  
- Sprint Review  
- Sprint Retrospective  

---

### 2.2 작업 수행 전략 (WBS)

| 단계 | 작업 항목 | 세부 내용 |
|------|----------|----------|
| 1 | 요구사항 분석 | 기능 정의, 사용자 요구 정리 |
| 2 | 시스템 설계 | 아키텍처 설계, DB 설계 |
| 3 | 인프라 구축 | AWS EC2, RDS, S3 설정 |
| 4 | 사용자 기능 개발 | 업로드/다운로드, 검색 기능 |
| 5 | 관리자 기능 개발 | 권한 관리, 사용자 관리 |
| 6 | 통합 테스트 | 기능 통합 및 오류 수정 |
| 7 | 배포 및 안정화 | 서비스 배포 및 모니터링 |

---

### 2.3 일정 계획 (Gantt 개념)

| 기간 | 작업 |
|------|------|
| 1~2주 | 요구사항 분석 및 설계 |
| 3~4주 | 인프라 구축 |
| 5~8주 | 핵심 기능 개발 |
| 9~10주 | 부가 기능 개발 |
| 11~12주 | 테스트 및 배포 |

**Sprint 운영**
- 2주 = 1 Sprint  
- 매일 Daily Scrum 진행  

---

### 2.4 자원 계획
- 개발 인원: 약 10명  
- 기능별 팀 구성  
- 클라우드 환경 활용  

---

### 2.5 일정 관리
- Jira 기반 Kanban 관리  
- 주간 회의 및 Sprint 회고  
- 일정 지연 시 우선순위 재조정

## 3. 조직 구성 (Organization)

### 3.1 팀 구조
- 중앙 관리 + 기능별 분산 구조 (Hybrid)

### 3.2 역할 및 책임
- PM: 일정 관리 및 총괄  
- PL: 시스템 설계 및 기술 방향성  
- 개발자: 기능 구현  
- QA: 테스트 및 품질 검증  
- 형상 관리자: 버전 관리  

---

## 4. 기술 관리 계획 (Technical Management)

### 4.1 버전 관리
Git 기반 브랜치 전략 적용

**브랜치 구조**
- `main`: 배포 버전  
- `develop`: 통합 개발  
- `feature/*`: 기능 개발  

---

### 4.2 코드 관리 정책
- Pull Request 기반 코드 리뷰  
- 승인 후 main 브랜치 병합  

---

### 4.3 형상 관리

**파일 명명 규칙**
- 문서: `문서명_버전_날짜`
  - 예: `Requirement_v1.0_20260429.docx`
- 브랜치: `feature/기능명`
  - 예: `feature/file-upload`

---
