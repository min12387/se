# 프로젝트 관리 계획서

---

## 1. 개요 (Overview)

### 1.1 문서 목적
* 본 문서는 Mini Drive 시스템 개발을 체계적으로 수행하기 위해 프로젝트의 일정, 인력 구성, 기술 운영 방식, 품질 기준 및 위험 대응 전략을 정의하는 것을 목적으로 한다.

### 1.2 프로젝트 개요
* Mini Drive는 조직 내 파일을 통합적으로 관리하고 협업을 지원하기 위한 클라우드 기반 파일 관리 시스템이다.  
* 대용량 파일의 업로드 및 다운로드를 안정적으로 처리할 수 있도록 설계되며, 사용자 간 파일 공유 및 접근 권한 설정 기능을 제공한다.  
* 또한 파일 검색 기능과 활동 기록을 통해 데이터 관리의 효율성과 추적 가능성을 향상시키는 것을 목표로 한다.

### 1.3 참조 문서
* Project Definition Document  
* Requirement Specification  
* Quality Attribute Document

## 2. 개발 계획 (Development Plan)

### 2.1 개발 방식
* 본 프로젝트는 반복적 개발 방식과 Agile 기반 접근을 적용하여 기능 단위로 시스템을 점진적으로 완성한다.  
* 각 개발 단계에서 기능을 검증하고 사용자 피드백을 반영하여 지속적으로 개선한다.

### 2.2 작업 수행 전략
* 요구사항 분석 및 설계를 통해 전체 시스템 구조를 정의한다.  
* 이후 인프라 구축 → 핵심 기능 개발 → 부가 기능 개발 순으로 진행한다.  
* 최종적으로 통합 테스트를 수행한 후 배포 및 안정화 단계를 거친다.

### 2.3 자원 계획
* 약 10명 내외의 개발 인력으로 팀을 구성하여 프로젝트를 수행한다.  
* 클라우드 환경을 활용하여 시스템의 확장성과 안정성을 확보한다.

### 2.4 일정 관리
* Jira를 활용하여 작업을 관리하고 Kanban 보드를 통해 진행 상황을 시각적으로 확인한다.  
* 정기적인 회의를 통해 일정 진행 상태를 점검하고 필요 시 계획을 조정한다.

## 3. 조직 구성 (Organization)

### 3.1 팀 구조
* 프로젝트는 중앙 관리 구조와 기능별 분산 구조를 혼합한 형태로 운영한다.  
* 주요 의사결정은 PM과 PL이 담당하며, 각 개발 파트는 기능 단위로 자율적으로 수행한다.

### 3.2 역할 및 책임
* PM: 전체 일정 관리 및 프로젝트 진행 총괄  
* PL: 시스템 구조 설계 및 기술 방향성 수립  
* 개발자: 기능 구현 및 시스템 개발 수행  
* QA: 테스트 수행 및 품질 검증  
* 형상 관리자: 코드 및 문서 버전 관리  

---

## 4. 기술 관리 계획 (Technical Management)

### 4.1 버전 관리
* 소스 코드는 Git을 기반으로 관리하며, 기능 단위로 브랜치를 분리하여 개발을 진행한다.  
* 각 기능은 독립적으로 개발된 후 통합되어 시스템에 반영된다.

### 4.2 형상 관리
* 문서 및 산출물은 일정한 규칙에 따라 날짜와 버전을 포함하여 관리한다.  
* 모든 자료는 GitHub 저장소 내 지정된 폴더에 저장하여 변경 이력을 관리한다.

### 4.3 코드 관리 정책
* 코드 변경 사항은 Pull Request를 통해 검토 과정을 거친다.  
* 검토 완료 후 main 브랜치에 병합하여 품질을 유지한다.

---

## 5. 품질 및 위험 관리 (Quality & Risk)

### 5.1 품질 목표
* 사용자 요청에 대해 빠른 응답 속도를 유지한다.  
* 안정적인 파일 업로드 및 다운로드 환경을 제공한다.  
* 보안성을 고려한 접근 제어 체계를 구축한다.

### 5.2 위험 관리
* 요구사항 변경에 대비하여 반복적인 개발 방식을 적용한다.  
* 서버 부하 발생 시 확장 가능한 구조를 통해 대응한다.  
* 보안 취약점 방지를 위해 정기적인 점검을 수행한다.  
* 일정 지연 발생 시 우선순위 조정을 통해 대응한다.
